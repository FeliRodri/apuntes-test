## 1. Introducción al Modelo Lógico

El proceso de diseño comienza transformando un **modelo conceptual** en uno **lógico**. Según el material de la USM, el modelo lógico es aquel que se genera al tomar uno conceptual e incorporarle las características propias de la clase de software donde se implementará[cite: 11].

<Aside type="caution" title="Nota del Profe">
  Felipe, recuerda que en esta etapa ya no somos "agnósticos" al tipo de
  tecnología. Si bien no elegimos el motor específico (como PostgreSQL), ya
  decidimos que usaremos la *clase* de software "Relacional"[cite: 11, 14].
</Aside>

### Tipos de Modelos Lógicos

Existen diversos modelos lógicos según la estructura de datos que soportan[cite: 11]:

- **Jerárquico** [cite: 12]
- **Reticular** [cite: 13]
- **Relacional** (Nuestro enfoque principal) [cite: 14]
- **Orientado al Objeto** [cite: 15]
- **Multidimensional** [cite: 17]

## 2. Terminología Básica (Hoja 4)

Para hablar en "relacional", debemos cambiar nuestro vocabulario técnico[cite: 22]:

| Término Conceptual | Término Relacional          | Definición                                                                |
| :----------------- | :-------------------------- | :------------------------------------------------------------------------ |
| Archivo / Entidad  | **Relación** [cite: 23]     | Tabla bidimensional donde el orden de las filas no importa[cite: 24, 25]. |
| Registro           | **Fila / Tupla** [cite: 25] | Cada entrada individual en la tabla.                                      |
| Atributo           | **Columna** [cite: 26]      | Contiene valores del mismo tipo para todos los registros.                 |
| Rango de valores   | **Dominio** [cite: 27]      | El conjunto de valores posibles para una columna específica.              |

<Aside type="tip" title="Concepto Clave: Unicidad">
  En el modelo relacional, **cada fila debe ser única**[cite: 31]. Esto se logra
  mediante la **Clave Primaria (PK)**, una columna que no permite valores
  repetidos[cite: 31].
</Aside>
--- ## 3. Estrategias de Diseño: Top-Down vs. Bottom-Up (Hoja 6)

Para obtener un modelo relacional, existen dos caminos principales:

- **Esquema Top-Down:** Se construye primero un modelo conceptual y luego se convierte en uno relacional aplicando reglas de transformación formales[cite: 37].
- **Esquema Bottom-Up:** Se integran modelos parciales obtenidos de la normalización de vistas (como reportes o salidas del sistema)[cite: 38].

<Aside type="note" title="Enfoque del Curso">
  Felipe, en esta unidad nos enfocaremos exclusivamente en el **Top-Down**. Es
  el estándar cuando diseñas un sistema desde cero, partiendo de los requisitos
  del negocio hacia la implementación técnica[cite: 3, 37].
</Aside>

## 4. El Ciclo de Vida del Diseño (Hoja 7)

El proceso sigue una jerarquía de dependencia tecnológica. Es fundamental distinguir qué partes dependen del motor de base de datos (SGBD) y cuáles no:

### Fase Independiente del SGBD

1.  **Recolección de Requisitos:** Entender qué necesita el usuario[cite: 48].
2.  **Diseño Conceptual:** Creamos el "Modelo Conceptual" (usualmente diagramas de clase UML para este curso)[cite: 52, 144].

### Fase Dependiente del SGBD

1.  **Diseño Lógico:** Aquí es donde transformamos ese mapa conceptual a tablas (Modelo Lógico)[cite: 56].
2.  **Diseño Físico:** Optimizamos para el hardware y software específico (índices, particiones)[cite: 61].
3.  **Implementación:** Código SQL real[cite: 71].

<Aside type="danger" title="¡Cuidado en el Examen!">
  Un error típico es intentar elegir el software (PostgreSQL, Oracle, etc.)
  antes de terminar el Diseño Conceptual. Según el flujo de la USM, la
  **Elección del SW** ocurre justo antes o durante el Diseño Lógico, no
  antes[cite: 74, 110].
</Aside>

## 5. Etapa 1: Recolección y Análisis (Hoja 8)

Antes de tirar una sola línea de código, debemos:

- Identificar grupos de usuarios y áreas de aplicación[cite: 81].
- Estudiar documentación existente: manuales, reportes y diagramas[cite: 82, 83].
- **Objetivo final:** Identificar las necesidades de información[cite: 78, 79].

## 6. Caso de Estudio: Campeonato de Fórmula 1 (Hojas 9-10)

Antes de aplicar reglas de transformación, debemos identificar los hechos clave del enunciado[cite: 88]:

### Análisis de Requisitos

- **Escudería y Pilotos:** Una escudería tiene al menos un piloto y puede tener varios; el piloto firma por una temporada[cite: 89, 90].
- **Países:** Las escuderías pertenecen a un país, y un país puede tener muchas escuderías[cite: 90, 91].
- **Automóviles:** Pertenecen a una escudería y se asignan a un piloto para una carrera específica si están disponibles[cite: 92, 93].
- **Carreras y Circuitos:** Muchas carreras ocurren en circuitos; un circuito puede tener varias carreras en distintas temporadas[cite: 99, 100].

### Modelo Conceptual (Diagrama de Clases UML)

Utilizaremos un diagrama para visualizar estas entidades antes de convertirlas a tablas:
